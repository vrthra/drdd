# ZipMin Benchmark Suite

Tools and artefacts for benchmarking `dd.zipmin` against the classic `dd.ddmin` XML delta-debugging workflow. The suite orchestrates predicate setup, runs both algorithms per input variant, and records wall-clock timings for comparison.

## Directory Map

- `scripts/bench_zipmin.py` – main harness that spawns BaseX-backed predicates, runs both algorithms, and writes a CSV summary under `runs/`.
- `scripts/generate_inputs.sh` – helper to regenerate `input.pick/<n>.xml` seed corpora for selected predicates via `scripts/cherry_pick`.
- `runs/` – dated result folders (e.g. `14-09-2025_18:12_git-77472dd/`) containing the CSV summary, per-run logs, and minimized outputs.

## Prerequisites

Follow the project-level setup from the repo root (`pip install -e .`). The benchmark harness additionally expects:

- Python 3.8+ (matching the editable install).
- Java 11+ plus `nc`/`netcat` for the BaseX wrapper; see `scripts/README.md` for details.
- Predicate assets under `predicates/xmlprocessor/` (already checked into the repo).

## Preparing Test Inputs

The harness reads inputs from `predicates/xmlprocessor/xml-*/input.pick/<variant>.xml`. Regenerate them when predicates change:

```bash
bash benchmark/scripts/generate_inputs.sh
```

The script creates five variants (`1..5`) for each listed predicate by invoking `scripts/cherry_pick` through the BaseX wrapper. Adjust the `case` loop or flags inside the script if you need different predicates, sizes, or variant counts.

## Running the Benchmark

Activate your virtualenv (or export `PYTHONPATH=src`) and invoke the harness:

```bash
python benchmark/scripts/bench_zipmin.py \
  --pred-root predicates/xmlprocessor \
  --variants 1,2,3,4,5 \
  --jobs 4
```

### Key options

- `--pred-root` – directory containing predicate subfolders (`xml-*`). Defaults to `predicates/xmlprocessor` relative to the repo.
- `--variants` – comma-separated variant indices to test (default `1..5`). Missing files are skipped with a warning.
- `--jobs` – level of parallelism; default is `max(1, cpu_count // 2)` to keep BaseX load manageable.
- `--output` – target CSV path. The parent folder determines where per-run artefacts are stored.

Each benchmark invocation creates a new folder under `runs/` named `<DD-MM-YYYY>_<HH:MM>_git-<short-sha>/` so multiple runs remain side-by-side. The harness streams stdout/stderr for each job into `runs/<run>/logs/` and prints progress updates on the console.

## Outputs

Inside each `runs/<run>/` directory you will find:

- `result.csv` – tabular summary with timestamps, wall-clock durations, algorithm/predicate metadata, input and minimized sizes, hash digests, reduction ratios, and oracle counts.
- `logs/` – stdout/stderr for every algorithm × predicate × variant run, useful for troubleshooting failures.
- `minimized/` – minimized XML artefacts written by each algorithm.
- Optional extras (e.g. `report.pdf`) may be generated by downstream analysis scripts.

The CSV header is constructed automatically from observed keys, so new metrics added to the harness will appear without further changes.

## Tips & Troubleshooting

- Make sure `scripts/minimize_xml` and related helpers are executable (`pip install -e .` handles their dependencies).
- Use `--jobs 1` if BaseX servers or predicates saturate your machine.
- Logs record the exact command and error conditions when a run fails; inspect the matching files under `runs/<run>/logs/`.
- To compare runs, diff the CSV files or load them into a notebook/BI tool for aggregate metrics.
